# CBUR

A component to backup/restore Kubernetes volumes, objects, pvc and NCS cluster data.

## Introduction

This chart bootstraps a CBUR deployment on a [Kubernetes](http://kubernetes.io) cluster using the [Helm](https://helm.sh) package manager.

## Prerequisites
  - Kubernetes 1.18+

## Pod Security Standards
Helm chart can be installed in namespace with `restricted` Pod Security Standards profile.

### Special cases
Helm chart needs to be installed in namespace with `privileged` Pod Security Standards profile on environment with any of below scenarios:  
  1. .Values.avamar.enabled and .Values.hostNetwork.avamar
Reason: hostNetwork and hostPorts are needed
  2. .Values.k8swatcher.enabled and .Values.k8swatcher.clusterBrEnabled
Reason: hostPath is needed to be used and container should run as root
  3. .Values.volumeType.glusterfs  
Reason: container should run as root
  4. .Values.istio.enabled and (not .Values.istio.cni.enabled)
Reason: Linux Capability "NET_ADMIN" and "NET_RAW" are needed and allowPrivilegeEscalation must be true for non-root user.
  5. (.Values.global.tls.enabled or .Values.tls.enabled) and .Values.tls.certsPath and (not .Values.tls.serverSecretRef.credentialName) and (not .Values.tls.clientSecretRef.credentialName)  
Reason: hostPath is needed

#### Mapping to gatekeeper constraints
All gatekeeper constraints please refer to [capabilities](https://github.com/open-policy-agent/gatekeeper-library/blob/master/library/pod-security-policy/capabilities)

1. open source Gatekeeper `K8sPSPCapabilities` and `K8sPSPAllowPrivilegeEscalationContainer` constraint need to be relaxed to allow the following additional privileges:
  ```yaml
          capabilities:
            add: [ "NET_ADMIN", "NET_RAW" ]
  ```
2. open source Gatekeeper `K8sPSPHostNetworkingPorts` constraint needs to be relaxed to allow hostNetwork
3. open source Gatekeeper `K8sPSPHostFilesystem` constraint needs to be relaxed to use hostPath
4. open source Gatekeeper `K8sPSPAllowedUsers` constraint needs to be relaxed to run As root 
 
## Installing the Chart

To install the chart with the release name `cbur-master`:
## note: to work with helm plugin(e.g. helm backup), release name need to be cbur-master.

```console
# helm install --name cbur-master --namespace default --set volumeType.glusterfs=false,replicas=1 csf-incubator/cbur

if celery is to use
# helm install --name cbur-master --namespace default --set volumeType.glusterfs=false,replicas=1,direct_redis.enabled=true csf-incubator/cbur

```

The command deploys CBUR on the Kubernetes cluster control hosts Currently only 1 replica supported. [configuration](#configuration) section lists the parameters that can be configured during installation.

> **Tip**: List all releases using `helm list`

## check the log at host.
find the cbur-master.. pods (e.g. cbur-master-667ddfddd4-lscjb ) by cmd:
helm status cbur-master

- check the pod stdout
  kubectl logs -f cbur-master-cbur-59d94c569d-ft2jq
  kubectl logs -f cbur-master-cbur-mycelery-77c8546447-9f4p5

- check the detailed CBUR logs
  kubectl exec cbur-master-cbur-mycelery-77c8546447-9f4p5 -n default -c cbur -- bash -c "tail -f /CBUR_REPO/log/BR.log"

- check the swagger server log
  kubectl exec cbur-master-cbur-59d94c569d-ft2jq            cat /var/log/gunicorn/run.log

- check the celery log
  kubectl exec cbur-master-cbur-mycelery-77c8546447-9f4p5   cat /var/log/celery/run.log

## Uninstalling the Chart

To uninstall/delete the `cbur-master` deployment:

```console
$ helm delete --purge cbur-master
```

## Test service

Verify cbur service are ready.
(note: servicePort is defined in values.yaml and default is 80.)

```console
$ curl -LS http://cbur-master-cbur.default.svc.cluster.local:80/v2/status/
```
Expected output:

```console
"Ready to go!"
```
